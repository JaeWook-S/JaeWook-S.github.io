Project Overview

I am currently participating in the RoboCup 2026 Humanoid Soccer League as a Team Leader, leading Inha-United, a joint team formed by four robotics labs at Inha University (SPARO, RCV, ARTEMIS, and RILS).

As a first-time RoboCup team, we had to build a collaborative robot system under a limited preparation period. I was responsible for Vision, system modularization, and the design of a Behavior Tree-based control architecture, focusing on improving the structural robustness and scalability of the robot software.

In addition, as the Team Leader, I managed the overall competition process, including the preparation of the Team Description Paper (TDP), team homepage, and GitHub repository, ensuring smooth participation from the qualification rounds through to the finals.

The team homepage and GitHub repository can be found here: Homepage | GitHub

Below, I describe the detailed technical approaches and preparation process for this project.

System Overview

Overall system architecture of the humanoid robot software pipeline.

Vision

The system processes RGB-D camera input to extract visual information and converts it into robot-relative perception data for localization and decision-making.

Localization

The localization module estimates the robot pose using odometry and a particle filter based on field landmarks.

Decision-Making

The decision-making module performs role assignment and behavior selection based on game state information from perception and localization modules.

Control

Robot motion control was performed using the Booster Robotics SDK, where pre-defined control policies were utilized to trigger corresponding low-level API commands for locomotion and actions.

Communication

UDP broadcast communication is used to share game state information from the game controller as well as robot pose, ball position, and pass signals among robots.

My Contribution - Vision

Camera Calibration

The transformation between the robot’s camera and head frame was estimated through calibration.

First, a commonly used checkerboard-based calibration method was applied.

As a result, projection accuracy was relatively high for nearby markers. However, as the distance increased, we observed a systematic error in which projected positions appeared closer than their actual locations.

To address this issue, we redesigned the calibration approach by leveraging marker positions that are pre-defined in the RoboCup field environment. Using this prior knowledge, we performed a marker-based calibration instead of relying on a checkerboard pattern.

This approach resulted in a more robust calibration that provides consistent transformation accuracy, especially for long-range observations in real-world conditions.

3D-to-2D Ground Projection

To utilize detected objects for decision-making and localization, image-space detections were transformed into the robot base coordinate frame.

For each detected object, we selected a representative point from the bounding box, either the center or the bottom-center point, depending on the object type. This point was then projected into 3D space using the camera calibration parameters.

Specifically, we applied the camera extrinsic parameters along with the head-to-base transformation to convert the point into the robot's base frame.

Considering that the RoboCup soccer field can be approximated as a 2D plane, we computed the intersection between the projection ray and the ground plane (z = 0). This intersection point was used as the final estimated position of the object in the robot coordinate system.

Depth Obstacle Detection

To detect obstacles that were not recognized by the object detection model, depth information was additionally utilized.

For real-time performance, point cloud downsampling was applied, and the space was divided into 20cm × 20cm grids. The number of point cloud points within each grid was counted, and obstacles were determined based on a predefined threshold.

If the number of points inside a grid exceeded the threshold, the corresponding region was classified as an obstacle.

Data Collection & Labeling

To build a robust training dataset, we collected publicly available data from the Bit-Bots team and Roboflow, resulting in approximately 16,000 real-world images.

However, these datasets did not include scenarios specific to our robot platform. To address this gap, we additionally collected a private dataset tailored to our hardware setup and conducted manual labeling.

To improve labeling efficiency, we leveraged a model pre-trained on the initial 16,000 images to perform first-pass inference. The predicted annotations were then refined and corrected, significantly reducing manual labeling effort.

My Contribution - Modularization

We developed our system based on the Booster Robotics demo code. Although components such as Vision, Communication, and the Behavior Tree (BT) framework were partially modularized, the core BT action logic itself was not properly modularized.

To improve code scalability and maintainability, we reorganized the BT structure by separating each action into independent modules.

This improvement made it much easier to add new actions and significantly simplified the development process. As a result, development became faster and more efficient, while code readability and reusability were also greatly improved.

My Contribution - Behavior Tree

Head Control

Head control is performed in two main situations: when the ball position is unknown, and when a detected ball is being tracked.

When the ball position is unknown, the head actively searches for the ball by moving its pitch and yaw in predefined patterns to cover a wider area. Once the ball is detected, the search action is immediately stopped, and the system transitions to the tracking phase.

During the tracking phase, the head adjusts its pitch and yaw so that the bottom-center point of the detected bounding box remains within a certain offset range around the center of the image frame. This ensures that the ball stays stably near the center of the camera view.

Chase Target Point

The chase target is determined based on the difference between the robot’s current orientation and the desired kick direction.

If the angular difference between the robot’s heading (theta) and the kick direction is below a certain threshold, the target is set to a point located at a fixed distance (dist) behind the ball. This point lies along the kick direction vector.

On the other hand, if the angular difference is large, a circular region centered at the ball with a radius of safeDist is considered. In this case, a tangent direction from the robot’s current position to the circle is computed, and the target is selected along this direction.

Kick Motion

The kick motion is executed by invoking a policy provided by Booster Robotics through the SDK. It takes the kick direction, the ball position in the robot coordinate frame, and the desired power as inputs, and performs vision-based dribbling and kicking.

The policy consists of an adjust phase, which refines and aligns the robot’s posture based on the ball position, and a kick phase, which executes the actual strike.
